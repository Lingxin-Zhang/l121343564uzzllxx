## OFEC Backend Contract

- `_ebch_backends.py`
- `_ebch_lut.py`
- `ebch.py`
- `scripts/reproduce_fig3_baseline.py`

`BCHBackend` requires:

- `encode(info) -> codeword`
- `decode(rx_base) -> (corrected_word, indicator)`
- `encode_many(info_batch) -> codeword_batch`
- `decode_many(rx_base_batch) -> (corrected_batch, indicators)`

Shapes and semantics:

- `encode`: `(239,) -> (255,)`
- `decode`: `(255,) -> ((255,), int)`
- `encode_many`: `(batch,239) -> (batch,255)`
- `decode_many`: `(batch,255) -> ((batch,255), (batch,))`
- indicator: `-1` failure, `0` no error, `1/2` one or two corrected BCH bits

`build_bch_backend()` returns an object backend unchanged, so
`EBCH256_239(backend=<BlockLUTBCHBackend>)` is enough to inject the external
backend.

## Matrix And Locator Convention

The official `SyndromeLUTBCHBackend` builds:

- `column_syndromes`: packed syndrome contribution per BCH bit position
- `column_syndrome_bits`: unpacked `(255,16)` GF(2) matrix
- `packed_to_bits`: packed syndrome to 16-bit vector table
- `decode_lut`, `decode_counts`, `decode_positions`: 0/1/2-error locator tables

The first 239 columns are generated by one-hot systematic
`galois.BCH(255,239)` encoding and taking the parity suffix. The final 16
columns are the identity. Thus `column_syndrome_bits` represents `H^T=[P;I]`,
and parity generation is `uP`.

The new backend imports OFEC's own `build_syndrome_lut_tables()` and reuses
those exact tables. It only replaces the online GF(2) linear map with
`PackedBlockLUTKernel.apply_many_packed`; the run uses `block_width=14`.
